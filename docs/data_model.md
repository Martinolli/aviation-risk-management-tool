# Data Model

## Overview

This document describes the initial data model concepts for the Aviation Risk Management Tool. It is not a final schema, but it should guide future PostgreSQL design and implementation tasks.

The model should support structured risk management, authority review, audit history, DOCX reporting, and LLM-assisted structuring.

## Core Entities

### User

Represents a person who can access the system.

Key fields may include:

- id
- name
- email
- active status
- role assignments
- created and updated timestamps

### Role

Defines permissions or permission groups for role-based access control.

Example roles may include:

- administrator
- risk creator
- risk owner
- action owner
- operational board member
- Risk Management Committee member
- Executive Safety Management Committee member
- auditor

### Authority Level

Represents the review level for a risk.

Baseline levels:

- Low Level: Operational Boards
- Middle Level: Risk Management Committee
- High Level: Executive Safety Management Committee

The middle and high levels are fixed and cannot be deleted.

### Board or Committee

Represents a specific reviewing body.

Default low-level boards:

- Flight Test Safety Committee - Operation
- Aircraft Safety Committee - Engineering Board
- Industrial Safety Committee - Quality, Manufacturing, Production, Supply Chain, OHSE

Low-level boards are configurable. Admin users may create, rename, archive, activate, or deactivate them. Middle and high-level committees are fixed.

Important fields may include:

- id
- name
- authority level
- active flag
- archived flag
- created and updated timestamps

### Risk Record

The central entity for a managed SMS risk.

Important fields may include:

- id
- risk number
- title
- Problem Description
- status
- current authority level
- assigned board or committee
- risk owner
- created by
- created, updated, closed, and archived timestamps
- active or archived flags

### Hazard

Represents a hazard identified for a risk record.

Important fields may include:

- id
- risk record id
- description
- source
- active or archived flag

### Central Event

Represents the central event around which causes and consequences are structured.

Important fields may include:

- id
- risk record id
- description
- rationale

### Cause

Represents a cause that may lead to the central event.

Important fields may include:

- id
- risk record id
- central event id
- description
- active or archived flag

### Consequence

Represents a consequence that may result from the central event.

Important fields may include:

- id
- risk record id
- central event id
- description
- severity relevance
- active or archived flag

### Risk Assessment

Represents an initial or residual assessment.

Important fields may include:

- id
- risk record id
- assessment type
- severity
- likelihood or probability
- risk rating
- rationale
- assessor
- assessed timestamp

Assessment type should distinguish initial risk from residual risk.

### Mitigation

Represents a mitigation or control intended to reduce risk.

Important fields may include:

- id
- risk record id
- description
- mitigation type
- owner
- due date
- status
- completion evidence
- active or archived flag

### Review Decision

Represents a committee or board decision.

Important fields may include:

- id
- risk record id
- board or committee id
- decision
- decision rationale
- decided by
- decided timestamp
- next required action

Decisions may include approve, reject, request rework, escalate, defer, or close.

### Monitoring Entry

Represents periodic monitoring or effectiveness review.

Important fields may include:

- id
- risk record id
- review date
- reviewer
- status summary
- effectiveness notes
- next review date

### Generated Report

Represents a generated DOCX report or report metadata.

Important fields may include:

- id
- risk record id
- report type
- generated by
- generated timestamp
- file reference or storage key
- report version or data snapshot reference

### LLM Suggestion

Represents advisory output from the LLM assistant.

Important fields may include:

- id
- risk record id
- source Problem Description version or reference
- suggestion payload
- model metadata
- created timestamp
- reviewed by
- review status
- accepted fields or rejection rationale

Accepted or modified LLM suggestions must be auditable.

### Audit Log Entry

Represents the immutable history of meaningful changes.

Important fields may include:

- id
- actor id
- timestamp
- entity type
- entity id
- action
- previous value
- new value
- reason or comment
- correlation id

Audit log entries should not be physically deleted.

## Deletion and Archival Rules

Important records must not be physically deleted. Use fields such as:

- active
- archived
- deactivated_at
- archived_at
- closed_at

Physical deletion should be limited to non-controlled temporary data where policy allows it.

## Historical Traceability

The data model must preserve historical meaning. For example:

- renamed boards should not make old decisions ambiguous
- inactive users should remain linked to past decisions
- changed assessments should remain reconstructable through audit history
- generated reports should be linked to their generation timestamp or data snapshot

## Reporting Considerations

DOCX report generation should read from controlled entities and include audit-relevant metadata. Future schema work should decide whether reports are regenerated on demand, stored as generated files, or both.
